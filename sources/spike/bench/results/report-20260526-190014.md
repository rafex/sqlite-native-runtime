# Benchmark Spike — MQTT → SQLite (Fase 0)

**Fecha**: 2026-05-26 (macOS, Apple M1, 8GB RAM)
**Broker**: Mosquitto 2.1.2 localhost
**Mensajes publicados**: 100 000 (QoS=1, payload ~120 bytes)
**Batch size**: 500 msgs
**Flush timeout**: 200 ms

> **Nota QoS-1**: paho-mqtt (publisher) confirma ~65 600 de 100 000 mensajes
> dentro del timeout de confirmación (flow control de ventana). Los tres spikes
> recibieron exactamente los mismos ~65 600 mensajes → comparación equitativa.

## Análisis

### Throughput

```
Rust throughput:          8 973 msg/s   (baseline)
Java native throughput:   4 099 msg/s   (45.7 % del baseline)
Java JAR throughput:      3 645 msg/s   (40.6 % del baseline)
```

**Java native = 45.7 % de Rust.** Umbral de decisión: 80 %.
→ Java está significativamente por debajo del umbral.

### Latencia de commit

Los commits de Rust son sub-milisegundo (<1 ms max_commit). Java tiene picos de
13-15 ms. Con tres colas de prioridad (HIGH/NORMAL/LOW) y flush de 50 ms para HIGH,
la latencia de commit es crítica. Rust cumple; Java tiene margen más ajustado.

### Startup

| | Startup |
|---|---|
| Rust | 33 ms |
| Java native | 263 ms |
| Java JAR | 473 ms |

En despliegues con reinicio frecuente (k8s rolling, fallos, reinicios de broker),
el startup de Rust (33 ms) es materialmente mejor que Java native (263 ms, 8×).

### RAM

Rust usa 8.6 MB RSS en idle. Java native usa 27 MB (3×). Java JAR usa 122 MB (14×).
En servidores edge con 256 MB RAM disponible, la diferencia es significativa.

### Distribución

- Rust: **1 binario estático** (SQLite bundled). `scp` + `chmod +x` = listo.
- Java native: binario + `libether_sqlite_jni_runtime.so`. Requiere coordinar versiones de la .so.
- Java JAR: JVM + .so + JAR. Más piezas.

---

## Decisión

**→ Spike C — Rust**

Aplicando el criterio de decisión de `WORKER_IDEA.md`:

```
throughput(Java native) = 45.7 % de throughput(Rust) < 80 %   ← condición 1 NO cumplida
lógica de dominio: routing O(1) + INSERT genérico               ← minimal
→ Rust (Spike C): binario único, footprint mínimo
```

Factores adicionales que refuerzan la elección:
1. **Latencia**: commit < 1 ms → HEAD/NORMAL/LOW priority queues todas cumplibles
2. **Startup**: 33 ms → restart rápido en producción
3. **Distribución**: 1 binario, cero dependencias de runtime
4. **RAM**: 8.6 MB idle → puede correr junto a otros procesos en edge nodes
5. **SQLite bundled**: no requiere coordinar versión de .so entre worker y librería Java

### ¿Qué se pierde con Rust?

| Aspecto | Impacto |
|---|---|
| Ecosistema Java | Bajo: el worker no tiene lógica de dominio compleja |
| Observabilidad JMX | Sustituible por métricas HTTP (prometheus endpoint) |
| Tooling familiar | Costo de aprendizaje moderado; rumqttc + rusqlite son maduros |
| Mantenimiento | Rust es más difícil de modificar para desarrolladores Java-first |

El tradeoff es aceptable porque el worker es intencionalmente **thin**: recibe, parsea
topic, inserta en `ingest_event`. La lógica de negocio vive en el consumidor de esa tabla.
